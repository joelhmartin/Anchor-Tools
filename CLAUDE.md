This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Anchor Tools is a modular WordPress plugin (`anchor-tools.php`, currently 3.9.x). Each feature is a self-contained module that users can enable/disable from Settings > Anchor Tools. The plugin uses GitHub-based updates via Plugin Update Checker (it reads the GitHub Releases published by the release workflow).

## Key Commands

**Source is raw PHP/CSS/JS** — no transpilation, bundling, or ES modules; jQuery only. Author and enqueue the **source** `.css` / `.js` files.

**Minified assets are generated by CI, never committed.** `*.min.css`, `*.min.js`, and `*.min.js.map` are produced by `bin/build-assets.mjs` (csso + terser), which `.github/workflows/release.yml` runs automatically on a tag push so the release ZIP ships minified. They are **gitignored — never commit or hand-edit a `.min.*` file**; edit the source and let the release build regenerate it. To generate locally if you need to:

```bash
npm install --no-save csso terser && node bin/build-assets.mjs
```

```bash
composer install          # PHP deps (required after clone)

# PHPUnit — needs the WP test library (bin/install-wp-tests.sh) + a MySQL the tests can reach
export WP_TESTS_DIR=/tmp/wordpress-tests-lib WP_CORE_DIR=/tmp/wordpress
composer test

# Playwright E2E on @wordpress/env (Docker WordPress)
npm install && npm run wp-env start
npm run env:seed          # idempotent fixtures -> e2e/.seed.json
npx playwright test
```

**This repo DOES have automated tests.** PHPUnit (`phpunit.xml.dist`, `tests/`, base `Anchor_Events_TestCase`) and Playwright E2E (`.wp-env.json`, `bin/e2e-seed.sh`, `e2e/`). Coverage is concentrated in the **events module**; most other modules are still untested — add tests when you touch them.

## Architecture

### Bootstrap Flow

1. `anchor-tools.php` loads Composer autoload, `.env`, and core classes from `includes/`
2. On `plugins_loaded` (priority 25), `anchor_tools_bootstrap_modules()` iterates the module registry
3. For each enabled module: run optional `setup` callback → `require_once` the module file → instantiate the class → run optional `loader` callback
4. Module registry is in `anchor_tools_get_available_modules()` (~line 2870 of `anchor-tools.php`)

### Module System

Every module lives in `anchor-{name}/anchor-{name}.php` and follows one of two patterns:

**CPT-based** (most modules): Register a custom post type, metaboxes, `save_post` handler, admin columns, and shortcode. Data stored in post_meta with a module prefix (e.g. `avg_*`, `up_*`). Examples: video_slider, universal_popups, mega_menu, code_snippets, ctm_forms, events_manager, store_locator, webinars.

**Settings-page-based**: Register an options page under Settings and store data in a single option. Examples: social_feed, shortcodes.

See `ADDING-MODULES.md` for the full module development guide.

### Core Classes (`includes/`)

- `Anchor_Schema_Admin` — Main settings page, API key management, module toggles. Settings stored in `anchor_schema_settings` option.
- `Anchor_Schema_Helper` — Schema.org types, OpenAI API wrapper, JSON-LD utilities.
- `Anchor_Schema_Render` — Outputs JSON-LD on `wp_head`.
- `Anchor_Reviews_Manager` — Google Reviews fetching/caching, `[anchor_reviews]` shortcode.
- `Anchor_Schema_Logger` — Debug logging to PHP `error_log` (enabled in settings).

## Conventions

- **Asset paths**: Always use `ANCHOR_TOOLS_PLUGIN_URL . 'anchor-{module}/assets/'` — never `plugin_dir_url(__FILE__)` inside a module (it resolves incorrectly).
- **Options**: Always pass `autoload=false` as the third argument to `update_option()`.
- **Text domain**: Use `'anchor-schema'` for all translatable strings.
- **Class naming**: `Anchor_{ModuleName}_Module` for non-namespaced modules, `\Anchor\{ModuleName}\Module` for namespaced ones.
- **AJAX actions**: Prefix with `anchor_{module}_` (e.g. `wp_ajax_anchor_video_slider_preview`).
- **Admin assets**: Only enqueue on relevant admin pages — check `$hook` and post type in `admin_enqueue_scripts`.
- **CPT registration**: Use `apply_filters('anchor_{module}_parent_menu', true)` for `show_in_menu` so it can be overridden.
- **JavaScript**: jQuery IIFE pattern `(function($) { ... })(jQuery);` — no ES modules.
- **CSS/JS versions**: Bump version string in `wp_enqueue_*()` calls when updating assets.
- **Minified assets**: Enqueue the **source** `.css`/`.js`. Never commit or edit `*.min.css` / `*.min.js` / `*.min.js.map` — they're gitignored and regenerated by `bin/build-assets.mjs` in the release workflow.
- **Frontend rendering**: Shortcode callbacks use `ob_start()` / `return ob_get_clean()`.

## Release Process

1. **Merge the work into `main` FIRST — never tag off a feature branch.** Doing so has repeatedly diverged `main` from what's actually released (e.g. 3.9.09–3.9.11 were tagged off a feature branch while `main` sat at 3.9.08, so `main` was missing shipped features and a naive release would have rolled them back).
2. Bump `Version:` in the plugin header of `anchor-tools.php`, on `main`.
3. Push `main`, then push the tag (`N.N.N`) — **tag from `main`**.
4. The tag push triggers `.github/workflows/release.yml`, which automatically runs `composer install --no-dev`, generates minified assets via `bin/build-assets.mjs`, builds the release ZIP, and creates the GitHub Release. **Do not build or upload the ZIP by hand.** Because the Action installs prod deps and minifies fresh from source, the committed `vendor/` and asset state does not affect the shipped ZIP.
5. Plugin Update Checker detects the new version in WP admin.
